# Results

This folder stores reproducible outputs generated by the benchmark scripts.

## Main Result Folders

| Folder | Purpose |
|---|---|
| `prompt_input_runs/` | Prompt-ready EER inputs generated from conceptual ground truth files. |
| `llm_runs/` | Raw LLM execution outputs, rendered prompts, responses, usage, cost, and manifests. |
| `normalization_runs/` | Normalized LLM outputs and flattened schema artifacts. |
| `evaluation_runs/` | Evaluation metrics, errors, mapping reports, and evaluation manifests. |
| `batch_runs/` | Batch execution manifests and logs. |
| `aggregate_runs/` | Aggregated tables used for analysis and paper results. |

## Aggregate Outputs

The aggregation script is:

`python scripts/aggregate_results.py`

It produces outputs under:

`results/aggregate_runs/<run_id>/`

Expected aggregate files:

| File | Purpose |
|---|---|
| `aggregate_run_summary.csv` | Main table with one row per evaluated run. |
| `aggregate_llm_runs.csv` | Summary of LLM execution runs. |
| `aggregate_normalization_summary.csv` | Summary of normalization runs. |
| `aggregate_component_metrics.csv` | Component-level metrics by run, mode, and component. |
| `aggregate_evaluation_errors_detail.csv` | Detailed false positive and false negative items. |
| `aggregate_error_counts.csv` | Error counts and distances by evaluation mode. |
| `aggregate_cost_quality.csv` | Quality metrics combined with cost and latency data. |
| `aggregate_by_model.csv` | Mean results grouped by model. |
| `aggregate_by_condition.csv` | Mean results grouped by condition. |
| `aggregate_by_dataset_complexity.csv` | Mean results grouped by dataset complexity. |
| `aggregate_by_dataset_condition.csv` | Mean results grouped by dataset and condition. |
| `aggregate_by_model_condition.csv` | Mean results grouped by model and condition. |
| `aggregate_manifest.json` | Reproducibility manifest for the aggregation run. |

## Analysis Use

The aggregate files support analyses of:

- model quality;
- prompt condition effects;
- schema complexity effects;
- strict vs. matched vs. alternative-aware evaluation;
- naming mismatch;
- valid alternative mappings;
- invalid mappings;
- component-level weaknesses;
- hallucination and omission patterns;
- cost-quality trade-offs.
